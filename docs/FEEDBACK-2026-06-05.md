## #1 — DS-`@import` вне `@layer`, конфликт приоритетов с потребителем

### Что наблюдаем

Все DS-CSS-файлы импортируются в потребительский CSS через `@import "..."` **без `@layer`**. По спецификации CSS Cascading Layers, правила вне любого слоя имеют **более высокий приоритет**, чем правила внутри любого `@layer`. Tailwind v4 кладёт свой preflight в `@layer base`. Если потребитель кладёт свои override'ы в `@layer base` (что естественный, идиоматический способ для Tailwind-проекта), они **проигрывают DS**.

### Конкретный кейс из Антиштрафа

Мы написали в потребительском `index.css`:

```css
@layer base {
  html { font-family: var(--text-body-family); }
  body { background: var(--bg-primary); color: var(--text-primary); }
  @media (max-width: 759px) {
    .app-shell > .sb:not(.sb--collapsed) { display: none; }
  }
}
```

Результат: `--text-body-family` правильно резолвился в Dreamkas Sans, но `html { font-family: ... }` не применялся (на `<body>` оставался Inter из `<style>` в index.html). После выноса правил **из `@layer base` на top-level CSS** — заработало. Аналогично с `.sb { display: none }` на мобайле: оно проигрывало DS `.sb { display: flex }`, потому что DS — вне слоя.

### Что предлагаем

Вариант A (рекомендуется): обернуть всё содержимое `design-system/index.css` в `@layer dreamkas { ... }`:

```css
@layer dreamkas {
  @import "./foundations/color/primitives.css";
  /* ... */
}
```

Тогда потребитель явно управляет приоритетами:

```css
@layer base, dreamkas, overrides;
@import "tailwindcss";
@import "./design-system/index.css";

@layer overrides {
  body { background: var(--bg-primary); } /* выигрывает у dreamkas */
}
```

Вариант B (менее предпочтительный): задокументировать в `README` ДС, что «потребитель должен писать override'ы вне `@layer`», и привести пример.

### Срочность

**Высокая** — сейчас любой проект на Tailwind v4 (а это популярный стек) наткнётся на это в первые часы интеграции. У нас ушло ~30 минут отладки, чтобы понять, что виноваты слои, а не пустой токен или Vite-баг.

## #2 — Нет токена `--bg-surface`

### Что наблюдаем

В `color/semantic.css` есть:

```css
--bg-primary: var(--color-neutral-0);  /* Bg_Primary — основной фон страницы */
--bg-secondary: var(--color-neutral-100);
--surface-default: var(--color-neutral-0);
--surface-raised: var(--color-neutral-0);
--surface-sunken: var(--color-neutral-100);
```

Имя `--bg-primary` логично для дизайнера, но потребитель-разработчик интуитивно тянется к `--bg-surface` (термин из Material/Tailwind v4/iOS). Я в спеке и плане сам прописал `var(--bg-surface)` — пришёл в код, получил прозрачный body (`rgba(0,0,0,0)`), потому что переменная не существует. CSS `var()` молча возвращает initial.

### Что предлагаем

- **Минимум:** добавить алиас в `color/semantic.css`:
  ```css
  --bg-surface: var(--bg-primary);  /* alias для общего web-словаря */
  ```
- **Или:** в README блок «Если ищешь токен по имени из другой системы…» — таблица соответствий (Material → DS, Tailwind → DS, и т.п.). Это снизит трение потребителей.

### Срочность

**Высокая** — это failure-mode, в котором всё «выглядит ОК» (нет ошибки CSS), но фон неправильный, и потребитель долго ищет причину.

## #3 — 5 «универсальных» компонентов есть только в Web/

### Что наблюдаем

Структура DS rev 2:

```
Web/Components/    Atoms: Avatar, Badge, Button, Input, Select, Selection, Tooltip   (7)
                  Molecules: Alert, Card, Chart, FormField, Pagination, Table, ...    (9)
Tablet/Components/ Atoms: Button, Input, Select, Selection                            (4)
                  Molecules: Alert, Chart, Pagination, Table, TableToolbar, Tabs, Toast (7)
Mobile/Components/ (то же что Tablet)
```

Avatar/Badge/Tooltip/Card/FormField есть только в Web/. Это правильное решение — у них размер/паттерн един под все платформы. Но потребитель про это не знает, пока не наткнётся.

В нашем агрегаторе мы попытались скоупить **все** компоненты по media-query на платформу. Сидим в tablet/mobile-диапазоне — нет ни Avatar, ни Badge. Sidebar пытается показать `<span class="avatar avatar--md">` — никакого CSS. Авария.

Пришлось вручную разделить: «вот эти 14 компонентов есть во всех платформах — скоупим по media; вот эти 5 живут только в Web и активны всегда».

### Что предлагаем

Опция A: завести в Tablet/Mobile **пустые re-export'ы**:

```css
/* Tablet/Foundations/Component tokens/avatar/tokens.css */
@import "../../../Web/Foundations/Component tokens/avatar/tokens.css";
```

Тогда потребитель может тупо одной media-query'ёй подключать ВСЕ компоненты для платформы — поведение единообразно.

Опция B: в `README` ДС вывести явный список «универсальные компоненты» (avatar/badge/card/form-field/tooltip) и сказать «их подключай без media-scope». Опубликовать рекомендуемый агрегатор.

Опция C: в Patterns/ положить готовый агрегатор `design-system/index.css` который потребитель просто копирует.

Мы у себя сделали Опцию B inline в плане. Это работает, но требует разобраться. Опция C — самый дружелюбный путь.

### Срочность

**Средняя** — обходится за час знания, но без знания страница ломается тихо.

## #4 — TTF вместо WOFF2

### Что наблюдаем

В `Foundations/typography/fonts/`:
- `Roboto-Variable.ttf` (~2.5MB)
- `Roboto-Italic-Variable.ttf` (~2.5MB)

WOFF2-эквиваленты обычно в 2× раз компактнее (~1.2MB). Для web — это весомая разница на первой загрузке.

### Что предлагаем

Положить и `.woff2`, и `.ttf` (как fallback). В `fonts.css` использовать `src: url(...woff2) format('woff2'), url(...ttf) format('truetype')`. Современные браузеры возьмут WOFF2, старые — TTF.

### Срочность

**Средняя** — функционально работает с TTF, просто медленнее.

## #5 — Tablet/Select: имя `select.v2.css`

### Что наблюдаем

`Tablet/Components/Atoms/Select/` содержит `select.v2.css`, а не `select.css` (есть пояснение в CLAUDE.md ДС: был баг кэша раздачи, спасло переименование). Это значит, что любой потребительский генератор импортов, который ищет `<comp>.css`, на Tablet/Select **не находит файла**.

Мы у себя в агрегаторе нашли это случайно через `find -name "*.css"`. План раньше предполагал `select.css`.

### Что предлагаем

- Стабилизировать на каноническом имени `select.css` (если кэш-баг уже не воспроизводится в актуальной раздаче).
- Если приходится оставить `.v2.css` — положить рядом `select.css` как `@import "./select.v2.css";` (одна строка, нулевая стоимость, скрипт-генератор работает).
- Опубликовать в `README` явный список «именованных исключений» — чтобы потребитель знал и обработал.

### Срочность

**Средняя** — ломает автоматизацию, но не функционал.

## #6 — Канон App Shell **существует**, но похоронен в preview

### Что наблюдаем

Изначально я думал, что DS не даёт App Shell как композит — есть только `<aside class="sb">` и `<header class="tb">` как изолированные организмы. И в первой версии Куска 1 написал собственные имена `.app-shell / .app-shell-main / .app-shell-content` с flex-layout.

Пользователь проекта-потребителя спросил «а почему вы не используете App Shell из ДС?» — пошёл искать ещё раз и нашёл: `preview/organism-app-shell-web.html`, строки 42–73:

```css
.app {
  display: grid;
  grid-template-columns: var(--sb-width) 1fr;
  grid-template-rows: 100%;
  height: 800px;
  background: var(--bg-secondary);
  overflow: hidden;
}
.app-main { display: flex; flex-direction: column; min-width: 0; }
.app-content { flex: 1; overflow-y: auto; padding: 20px; }
```

И структура:

```html
<div class="app">
  <aside class="sb">...</aside>
  <div class="app-main">
    <header class="tb">...</header>
    <div class="app-content">...</div>
  </div>
</div>
```

Канон есть, имена есть, всё хорошо. Но он живёт **внутри `<style>`-блока demo-страницы** preview-карточки, а не как опубликованный CSS-файл в `Web/Components/Organisms/AppShell/app.css`. У потребителя возникает иллюзия отсутствия — он смотрит структуру `Web/Components/Organisms/` и видит только `Sidebar/`, `Topbar/`, `Modal/`.

### Что предлагаем

Опция A (рекомендуется): извлечь App Shell как полноценный организм:

```
Web/
  Foundations/Component tokens/app/tokens.css   ← новый файл с --app-bg, --app-padding и т.п.
  Components/Organisms/AppShell/app.css         ← .app, .app-main, .app-content
```

После этого preview-карточка может ссылаться на `app.css` через `<link>`, как все остальные превью.

Опция B: положить в `Web/Patterns/Navigation/app-shell.html` готовый сниппет с минимальным CSS, чтобы потребитель копировал не из preview, а из явного «Patterns»-файла.

### Маленький нюанс по grid-template-columns

В preview сейчас `grid-template-columns: var(--sb-width) 1fr`. Если Sidebar свёрнут (`.sb--collapsed` → `width: var(--sb-width-collapsed)`), grid-колонка всё равно `var(--sb-width)` — между Sidebar и контентом появится пустое место. У себя я заменил на `auto 1fr` — колонка реагирует на фактическую ширину `<aside class="sb">` и при collapse сжимается корректно. Рекомендуем DS зафиксировать `auto 1fr` в канон.

### Срочность

**Средняя** — функционально не блокирует (мы написали аналог с теми же именами и работает), но именно из-за неявности канона потребитель легко уходит в самопис. И при следующем апдейте DS у нас разъедутся структуры.

## #9 — Коллизия имён `--space-N` (DS) и `--spacing-N` (Tailwind v4)

### Что наблюдаем

DS экспонирует spacing-токены в виде:

```css
--space-2:  2px;
--space-4:  4px;
--space-8:  8px;
--space-16: 16px;
--space-24: 24px;
--space-32: 32px;
/* ... в пикселях, семантический шаг */
```

Tailwind v4 экспонирует свои:

```css
--spacing-2:  0.5rem;   /* 8px  */
--spacing-4:  1rem;     /* 16px */
--spacing-8:  2rem;     /* 32px */
--spacing-16: 4rem;     /* 64px */
/* ... в rem, шаг 0.25rem */
```

Имена с одним и тем же N означают **разные величины** (DS `--space-4` = 4px, Tailwind `--spacing-4` = 16px). Это footgun: интуитивная мысль «давайте сделаем мост `--spacing-N → var(--space-N)` чтобы Tailwind утилиты ходили в DS-шкале» **тихо ломает весь существующий код**.

### Конкретный кейс из Антиштрафа

В первой версии нашего `index.css` мы написали:

```css
@theme {
  --spacing-4: var(--space-4);
  --spacing-8: var(--space-8);
  /* ... */
}
```

Думали: «прокидываем DS-шкалу в Tailwind». Результат: каждый `p-4` в проекте (изначально 16px) стал **4px**. Каждый `p-8` (32px) стал 8px. Каждый `gap-6` стал 0 (потому что я не объявил `--spacing-6`). Все непокращенные страницы (Dashboard, ViolationDetails, Settings и т.д.) визуально схлопнулись — padding'и пропали, chart'ы наезжали на лейблы, hint-панели сидели впритык на колонки данных.

Заметили только когда пользователь прислал скриншот ViolationDetails с накладкой подсказки на KPI-колонку. До этого думали, что pages выглядят как раньше.

### Что предлагаем

Опция A (рекомендуется): **переименовать DS-токены** в `--ds-space-N` / `--ds-radius-N` / etc. Это раз и навсегда устраняет коллизию. Любой потребитель видит «о, это DS-namespace» и не путает с Tailwind/чужими шкалами.

Опция B: оставить имена как есть, но в `Foundations/spacing/README.md` и `AUDIT.md` **жирно предупредить**: «Имя `--space-N` НЕ алиас Tailwind `--spacing-N`. DS в пикселях, Tailwind в rem. НЕ ДЕЛАЙТЕ `--spacing-N: var(--space-N)`».

Опция C: переехать DS-spacing в rem-шкалу (`--space-1 = 0.25rem`, `--space-2 = 0.5rem`...). Тогда имена и значения как у Tailwind. Но это противоречит дизайнерской семантике (DS подаёт «4px / 8px / 12px» как осмысленные шаги, а не «1/16 / 1/8 / 3/16 rem»).

### Срочность

**Высокая.** Это failure-mode, в котором проект на Tailwind v4 (популярный стек) при «правильно выглядящем» мосте ломается без CSS-ошибок. И это не та поломка, которая видна в DevTools «вот этот файл не загрузился» — это семантическая поломка значений. Я лично потерял на этом ~2 часа от внедрения до диагностики (нашли только когда пользователь прислал скрин).

## #10 — Механизм дистрибуции для GitHub-импорта

### Стратегический контекст

Цель Dreamkas — использовать эту ДС на **всех продуктах**: антиштраф, касса (POS), backoffice, отчёты, мониторинг штрафов и т.д. План — «импортировать из GitHub и применять». Сейчас механизма для этого **нет**: репо `cooljekee/DreamkasDesignSystem` хранит исходники, но в README не описано, как их подключить из консьюмерского `package.json`.

### Что наблюдаем

Каждый продукт сам решает, как «забрать ДС»:
- Один сделает `git clone` в sibling-папку и `cp -r`.
- Другой добавит `git submodule`.
- Третий повторит мою историю: ручная копия 388 файлов в `design-system/` с агрегатором.

Через полгода у каждого продукта **своя ревизия** ДС (один на rev 2, другой на rev 2 + 3 локальных правки, третий уже скопировал rev 3). Расхождения нарастают.

### Что предлагаем

**Опция A (рекомендуется) — npm-пакет с git-installable URL:**

В корне репо ДС:

```json
{
  "name": "@dreamkas/ds",
  "version": "2.0.0",
  "files": ["rev 2/Foundations", "rev 2/Web", "rev 2/Tablet", "rev 2/Mobile", "rev 2/index.css"],
  "exports": {
    "./index.css": "./rev 2/index.css",
    "./foundations/*": "./rev 2/Foundations/*",
    "./web/*": "./rev 2/Web/*",
    "./tablet/*": "./rev 2/Tablet/*",
    "./mobile/*": "./rev 2/Mobile/*"
  }
}
```

Плюс готовый `rev 2/index.css` (агрегатор) — внутри пакета. Потребитель:

```bash
npm install "github:cooljekee/DreamkasDesignSystem#rev-2"
```

В своём `index.css`:

```css
@import "@dreamkas/ds/index.css";
```

Один тег версии — одна копия ДС. Обновление: `npm update @dreamkas/ds`.

**Опция B — git submodule:** для тех продуктов, где npm не используется.

```bash
git submodule add https://github.com/cooljekee/DreamkasDesignSystem.git design-system
```

Положить в README ДС готовый snippet'ы для обеих опций.

**Опция C — published npm package** (через npm.js): максимально удобно, но требует CI/CD-конвейера для публикации (можно сделать через GitHub Actions, тег = автоматический `npm publish`).

### Срочность

**Высокая для масштабирования.** Без этого «использовать ДС на всех продуктах» = ручной копи-паст в каждом проекте, что превращает 5+ продуктов в 5+ расходящихся форков. Это противоречит самой цели иметь ДС.

## #11 — Нет React/Vue/Svelte-обёрток

### Что наблюдаем

ДС поставляет HTML+CSS. Все продукты Dreamkas (антиштраф, касса, backoffice…) — на React (или похожем фреймворке). Каждый продукт пишет:

```tsx
// своя реализация AppShell, Sidebar, Topbar в каждом проекте
const Sidebar = ({ items, profile, collapsed, ... }) => (
  <aside className={collapsed ? 'sb sb--collapsed' : 'sb'}>
    {/* ... ручная разметка по канону, ручной toggle, ручной a11y ... */}
  </aside>
);
```

И в каждом продукте:
- Свой `useSidebarOpen` хук с breakpoint-логикой
- Свои focus-trap'ы для drawer'а
- Своя клавиатурная навигация
- Свои ARIA-роли
- Свои опечатки в `className`

Результат: 5 продуктов = 5 разных Sidebar'ов с одним и тем же визуалом, но разным поведением.

### Что предлагаем

Опубликовать `@dreamkas/react` (минимум на старте):

```tsx
import { AppShell, Sidebar, Topbar, Button, Input, FormField, Table, Modal } from '@dreamkas/react';

export function App() {
  return (
    <AppShell
      sidebar={<Sidebar items={NAV_ITEMS} profile={profile} />}
      topbar={<Topbar title="Товары" search={search} />}
    >
      <PageContent />
    </AppShell>
  );
}
```

- Компоненты внутри подключают CSS из `@dreamkas/ds` (peer dependency).
- Логика toggle/breakpoint/a11y/focus-trap — **один раз внутри пакета**.
- TypeScript типы — встроены.
- Версия `@dreamkas/react@2.0.0` соответствует `@dreamkas/ds@2.0.0`.

Этот шаг превращает интеграцию из «положить файлы + написать ~500 строк» в `npm install + import {AppShell}`.

### Срочность

**Высокая для масштабирования.** На 1 продукте без обёрток мы переживём (что и сделали). На 5+ продуктах без обёрток = неизбежный дрейф реализаций.

## #12 — Нет CHANGELOG между ревизиями

### Что наблюдаем

Когда выйдет rev 3, мне (как потребителю rev 2) нужно знать:
- Какие токены переименованы или удалены?
- Какие классы стали другими?
- Какие компоненты добавлены / удалены?
- Какие breaking changes требуют ручной правки в потребительских проектах?

Сейчас в репо: только git log. Каждый продукт диффает руками между ревизиями, тратит время на расследование.

### Что предлагаем

В корне репо ДС — `CHANGELOG.md`. Формат [Keep a Changelog](https://keepachangelog.com):

```markdown
## [rev 3] — 2026-09-01

### Breaking
- Переименован `--bg-primary` → `--bg-page`. Codemod: `npx @dreamkas/codemod rev2→rev3`
- Удалён `.btn--muted` — используй `.btn--tertiary` с `is-muted`

### Added
- App Shell как опубликованный CSS-файл: `Web/Components/Organisms/AppShell/app.css`
- Токен `--bg-surface` как алиас `--bg-page` (для consumer compatibility)

### Changed
- Sidebar: на mobile добавлен `.sb-bottom` вариант (bottom-nav)

### Fixed
- Опечатка в `--text-body-sm-letter-spacing`
```

Релизы тегаются: `git tag rev-3 && git push origin rev-3`.

### Срочность

**Высокая для масштабирования.** Без CHANGELOG продукты будут оставаться на старых ревизиях навсегда (страшно обновлять без понимания что сломается).

## #13 — Нет migration guide / codemods

### Что наблюдаем

Прямое следствие #12. Даже когда CHANGELOG есть («переименован `--bg-primary` → `--bg-page`»), потребитель идёт по каждому продукту вручную, grep'ом, заменяет.

5 продуктов × 100+ файлов на продукт × 5 переименований за ревизию = ~2500 ручных правок. Никто не будет это делать. Никто не обновится.

### Что предлагаем

Каждое breaking-изменение сопровождать codemod-скриптом:

```
rev-2-to-rev-3/
  README.md           — что меняется и как запустить
  codemod.mjs         — jscodeshift / sed-based; пробегается по проекту
  smoke-test.sh       — после запуска проверяет, что dev-сервер стартует
```

Потребитель:

```bash
npx @dreamkas/codemod rev2-to-rev3 ./src
```

Тулинг для codemod: `jscodeshift` для JSX/TSX, `postcss` или `sed` для CSS, `regex` для JSON.

Альтернатива при простых правках — раздел в CHANGELOG с готовыми `sed`-командами:

```bash
# Под Mac/Linux
find src -type f \( -name "*.tsx" -o -name "*.css" \) \
  -exec sed -i '' 's/--bg-primary/--bg-page/g' {} \;
```

### Срочность

**Высокая для масштабирования.** Без codemods продукты «застрянут» на rev 2. ДС перестанет эволюционировать в синхроне с продуктами.

## #14 — Нет TypeScript-определений

### Что наблюдаем

Я в JSX пишу `className="sb-item"` и `style={{ color: 'var(--bg-brand-default)' }}`. Если опечатался — `sb-iten`, `--bg-brand-defalt` — компилятор не ловит. Узнаю визуально, через час, при просмотре скриншота.

### Что предлагаем

Сгенерировать `.d.ts` из CSS — экспорт строковых литералов:

```typescript
// @dreamkas/ds/types.d.ts
export type DSClassName =
  | "sb" | "sb--collapsed" | "sb-logo" | "sb-logo-mark" | "sb-logo-text"
  | "sb-nav" | "sb-group" | "sb-item" | "sb-item-icon" | "sb-item-label"
  | "sb-item-badge" | "sb-footer" | "sb-profile" | "sb-profile-info"
  | /* ... весь список ... */ ;

export type DSToken =
  | "--bg-primary" | "--bg-secondary" | "--surface-default"
  | "--text-primary" | "--text-secondary"
  | "--bg-brand-default" | "--bg-brand-hover"
  | /* ... */ ;
```

Помощник для consumer'а:

```typescript
function cn(...classes: DSClassName[]): string {
  return classes.join(" ");
}
function dsToken(name: DSToken): string {
  return `var(${name})`;
}

// Использование с автокомплитом и проверкой опечаток
<aside className={cn("sb", collapsed && "sb--collapsed")} />
<div style={{ color: dsToken("--bg-brand-default") }} />
```

Скрипт-генератор `.d.ts`: пройтись по всем `.css`-файлам, вытащить `class*` селекторы и `--var-name` declarations, отдать как union-типы.

### Срочность

**Средняя.** Без типов жить можно, опечатки ловить через визуальное ревью. Но цена включения копеечная и польза постоянная.

## #15 — Нет accessibility-чеклиста на компонент

### Что наблюдаем

Sidebar — какая клавиатурная навигация? `Tab` или стрелки между пунктами? Какие ARIA-роли (`role="navigation"` на `<nav>`, `aria-current="page"` на активном `.sb-item.is-active`)? Что с focus-trap'ом, когда `.sb-overlay` открыт drawer'ом на mobile?

Modal — где `aria-modal="true"`, `role="dialog"`, `aria-labelledby`? Focus-trap внутри модала? Esc закрывает? Возврат фокуса на trigger при закрытии?

Tabs — стрелки между вкладками? `role="tablist"`, `role="tab"`, `aria-selected`?

Сейчас всё это — в голове у дизайнера или в head у потребителя. Каждый продукт пишет свою a11y-обёртку. Результат: антиштраф проходит axe-аудит, касса не проходит, отчёты частично — расхождения.

### Что предлагаем

В каждой папке компонента `Web/Components/<Comp>/` положить `A11Y.md`:

```markdown
# Sidebar — accessibility

## ARIA
- `<aside class="sb">` — `aria-label="Главная навигация"`
- `<nav class="sb-nav">` — без атрибутов, нативная семантика
- `.sb-item` — `role="link"` или `<a href>`; активный — `aria-current="page"`
- `.sb-item.is-disabled` — `aria-disabled="true"`, не убирать из DOM
- `.sb-item-badge` — `aria-label="N непрочитанных"` если число важно

## Keyboard
- Tab — между пунктами по порядку DOM
- Enter/Space на `.sb-item` — активация
- На mobile (`.sb-overlay`): Esc — закрыть drawer, фокус вернуть на бургер

## Focus trap
- На mobile drawer'е — да, внутри `.sb-overlay`
- На desktop — нет, естественный flow
```

Опционально — playwright + `@axe-core/playwright` test, который потребитель копирует в свой CI:

```typescript
import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

test('Sidebar passes a11y', async ({ page }) => {
  await page.goto('/');
  const results = await new AxeBuilder({ page }).include('aside.sb').analyze();
  expect(results.violations).toEqual([]);
});
```

### Срочность

**Средняя.** Без a11y-документации можно работать, но качество продуктов окажется неоднородным. В Enterprise UI (которым является Dreamkas backoffice) a11y часто требуется по контракту.

## Что ДС делает ОТЛИЧНО (для баланса)

- **3-слойная архитектура токенов (primitive → semantic → component)** — реально удобно. Имена семантического слоя интуитивны (`--text-primary`, `--border-focus`, `--status-danger-bg`).
- **Платформенный скоупинг через media-query на `@import`** работает идеально, как только понятно про universal-компоненты. DS-файлы НЕ модифицируются — это огромный win.
- **Sidebar и Topbar CSS-классы** — чистые, без лишнего. `.sb--collapsed`, `.sb-item.is-active`, `.tb-banner` — всё на своих местах. После того, как мы написали React-компоненты по канону, всё «зажглось» сразу.
- **Brand `#0097A7`** теплый и хорошо отбивается на нейтральном фоне. Лого-mark выглядит уместно и в развёрнутом, и в свёрнутом Sidebar.
- **Component tokens с `var()` на semantic/primitive** — позволяет менять размеры через токены, не трогая компонентный CSS. На tablet/mobile это и даёт работать одним media-query.
