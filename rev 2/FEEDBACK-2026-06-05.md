**От:** проект «Антиштраф» (React 19 + Vite + Tailwind v4)
**Куда:** куратор ДС, скилл «Софонов Дизайнер»

## #1 — DS-`@import` вне `@layer`, конфликт приоритетов с потребителем

### Что наблюдаем

Все DS-CSS-файлы импортируются в потребительский CSS через `@import "..."` **без `@layer`**. По спецификации CSS Cascading Layers, правила вне любого слоя имеют **более высокий приоритет**, чем правила внутри любого `@layer`. Tailwind v4 кладёт свой preflight в `@layer base`. Если потребитель кладёт свои override'ы в `@layer base` (что естественный, идиоматический способ для Tailwind-проекта), они **проигрывают DS**.

### Конкретный кейс из Антиштрафа

Мы написали в потребительском `index.css`:

```css
@layer base {
  html { font-family: var(--text-body-family); }
  body { background: var(--bg-primary); color: var(--text-primary); }
  @media (max-width: 759px) {
    .app-shell > .sb:not(.sb--collapsed) { display: none; }
  }
}
```

Результат: `--text-body-family` правильно резолвился в Dreamkas Sans, но `html { font-family: ... }` не применялся (на `<body>` оставался Inter из `<style>` в index.html). После выноса правил **из `@layer base` на top-level CSS** — заработало. Аналогично с `.sb { display: none }` на мобайле: оно проигрывало DS `.sb { display: flex }`, потому что DS — вне слоя.

### Что предлагаем

Вариант A (рекомендуется): обернуть всё содержимое `design-system/index.css` в `@layer dreamkas { ... }`:

```css
@layer dreamkas {
  @import "./foundations/color/primitives.css";
  /* ... */
}
```

Тогда потребитель явно управляет приоритетами:

```css
@layer base, dreamkas, overrides;
@import "tailwindcss";
@import "./design-system/index.css";

@layer overrides {
  body { background: var(--bg-primary); } /* выигрывает у dreamkas */
}
```

Вариант B (менее предпочтительный): задокументировать в `README` ДС, что «потребитель должен писать override'ы вне `@layer`», и привести пример.

### Срочность

**Высокая** — сейчас любой проект на Tailwind v4 (а это популярный стек) наткнётся на это в первые часы интеграции. У нас ушло ~30 минут отладки, чтобы понять, что виноваты слои, а не пустой токен или Vite-баг.

## #2 — Нет токена `--bg-surface`

### Что наблюдаем

В `color/semantic.css` есть:

```css
--bg-primary: var(--color-neutral-0);  /* Bg_Primary — основной фон страницы */
--bg-secondary: var(--color-neutral-100);
--surface-default: var(--color-neutral-0);
--surface-raised: var(--color-neutral-0);
--surface-sunken: var(--color-neutral-100);
```

Имя `--bg-primary` логично для дизайнера, но потребитель-разработчик интуитивно тянется к `--bg-surface` (термин из Material/Tailwind v4/iOS). Я в спеке и плане сам прописал `var(--bg-surface)` — пришёл в код, получил прозрачный body (`rgba(0,0,0,0)`), потому что переменная не существует. CSS `var()` молча возвращает initial.

### Что предлагаем

- **Минимум:** добавить алиас в `color/semantic.css`:
  ```css
  --bg-surface: var(--bg-primary);  /* alias для общего web-словаря */
  ```
- **Или:** в README блок «Если ищешь токен по имени из другой системы…» — таблица соответствий (Material → DS, Tailwind → DS, и т.п.). Это снизит трение потребителей.

### Срочность

**Высокая** — это failure-mode, в котором всё «выглядит ОК» (нет ошибки CSS), но фон неправильный, и потребитель долго ищет причину.

## #3 — 5 «универсальных» компонентов есть только в Web/

### Что наблюдаем

Структура DS rev 2:

```
Web/Components/    Atoms: Avatar, Badge, Button, Input, Select, Selection, Tooltip   (7)
                  Molecules: Alert, Card, Chart, FormField, Pagination, Table, ...    (9)
Tablet/Components/ Atoms: Button, Input, Select, Selection                            (4)
                  Molecules: Alert, Chart, Pagination, Table, TableToolbar, Tabs, Toast (7)
Mobile/Components/ (то же что Tablet)
```

Avatar/Badge/Tooltip/Card/FormField есть только в Web/. Это правильное решение — у них размер/паттерн един под все платформы. Но потребитель про это не знает, пока не наткнётся.

В нашем агрегаторе мы попытались скоупить **все** компоненты по media-query на платформу. Сидим в tablet/mobile-диапазоне — нет ни Avatar, ни Badge. Sidebar пытается показать `<span class="avatar avatar--md">` — никакого CSS. Авария.

Пришлось вручную разделить: «вот эти 14 компонентов есть во всех платформах — скоупим по media; вот эти 5 живут только в Web и активны всегда».

### Что предлагаем

Опция A: завести в Tablet/Mobile **пустые re-export'ы**:

```css
/* Tablet/Foundations/Component tokens/avatar/tokens.css */
@import "../../../Web/Foundations/Component tokens/avatar/tokens.css";
```

Тогда потребитель может тупо одной media-query'ёй подключать ВСЕ компоненты для платформы — поведение единообразно.

Опция B: в `README` ДС вывести явный список «универсальные компоненты» (avatar/badge/card/form-field/tooltip) и сказать «их подключай без media-scope». Опубликовать рекомендуемый агрегатор.

Опция C: в Patterns/ положить готовый агрегатор `design-system/index.css` который потребитель просто копирует.

Мы у себя сделали Опцию B inline в плане. Это работает, но требует разобраться. Опция C — самый дружелюбный путь.

### Срочность

**Средняя** — обходится за час знания, но без знания страница ломается тихо.

## #4 — TTF вместо WOFF2

### Что наблюдаем

В `Foundations/typography/fonts/`:
- `Roboto-Variable.ttf` (~2.5MB)
- `Roboto-Italic-Variable.ttf` (~2.5MB)

WOFF2-эквиваленты обычно в 2× раз компактнее (~1.2MB). Для web — это весомая разница на первой загрузке.

### Что предлагаем

Положить и `.woff2`, и `.ttf` (как fallback). В `fonts.css` использовать `src: url(...woff2) format('woff2'), url(...ttf) format('truetype')`. Современные браузеры возьмут WOFF2, старые — TTF.

### Срочность

**Средняя** — функционально работает с TTF, просто медленнее.

## #5 — Tablet/Select: имя `select.v2.css`

### Что наблюдаем

`Tablet/Components/Atoms/Select/` содержит `select.v2.css`, а не `select.css` (есть пояснение в CLAUDE.md ДС: был баг кэша раздачи, спасло переименование). Это значит, что любой потребительский генератор импортов, который ищет `<comp>.css`, на Tablet/Select **не находит файла**.

Мы у себя в агрегаторе нашли это случайно через `find -name "*.css"`. План раньше предполагал `select.css`.

### Что предлагаем

- Стабилизировать на каноническом имени `select.css` (если кэш-баг уже не воспроизводится в актуальной раздаче).
- Если приходится оставить `.v2.css` — положить рядом `select.css` как `@import "./select.v2.css";` (одна строка, нулевая стоимость, скрипт-генератор работает).
- Опубликовать в `README` явный список «именованных исключений» — чтобы потребитель знал и обработал.

### Срочность

**Средняя** — ломает автоматизацию, но не функционал.

## #6 — Нет канонического композита App Shell

### Что наблюдаем

DS даёт `<aside class="sb">` (Sidebar) и `<header class="tb">` (Topbar) — обоих как **изолированные организмы**. Layout-обёртка, объединяющая их в боковую панель + верхнюю плашку + контент-зону, в ДС не описана.

Мы у себя написали:

```html
<div class="app-shell">
  <aside class="sb">...</aside>
  <div class="app-shell-main">
    <header class="tb">...</header>
    <main class="app-shell-content">...</main>
  </div>
</div>
```

С CSS:

```css
.app-shell { display: flex; min-height: 100vh; width: 100%; }
.app-shell-main { flex: 1; min-width: 0; display: flex; flex-direction: column; }
.app-shell-content { flex: 1; overflow-y: auto; overflow-x: hidden; }
/* mobile: скрыть Sidebar в потоке */
@media (max-width: 759px) {
  .app-shell > .sb:not(.sb--collapsed) { display: none; }
}
```

Это, по сути, базовая «приёмочная» история. Хорошо бы поднять её на уровень ДС.

### Что предлагаем

Завести в `Web/Patterns/Navigation/` файл `app-shell.html` с готовым каркасом + CSS, чтобы потребитель копировал и не изобретал. Это согласуется с упоминанием в `README` про «композиты App Shell (Sidebar + Topbar)».

### Срочность

**Низкая** — фича, не блокер.

## #7 — Нет публикуемых правил для `body`/`html`

### Что наблюдаем

DS даёт токены `--text-body-family`, `--bg-primary`, `--text-primary`. Но НЕ даёт правил, которые их прикладывают к `body`/`html`. Это сделано намеренно (ДС не диктует, как устроен корень страницы потребителя) — справедливо. Но в результате каждый потребитель пишет похожий boilerplate:

```css
html { font-family: var(--text-body-family); }
body { background: var(--bg-primary); color: var(--text-primary); }
```

### Что предлагаем

Опционально (с явным комментарием «подключать, если хочешь дефолты ДС на корне страницы») завести `foundations/base.css`. Потребитель решает сам, подключать или нет.

### Срочность

**Низкая** — boilerplate терпимый.

## #8 — Нет описания паттерна Mobile drawer в `sidebar.css`

### Что наблюдаем

`sidebar.css` содержит `.sb-overlay` и `.sb-bottom`-варианты, но в комментарии-канон в начале файла описано только desktop-поведение. Поведение «< 760: Sidebar скрыт из потока, открывается оверлеем по бургеру» **неявное** — потребитель находит, читая CSS-классы.

### Что предлагаем

В шапку `sidebar.css` дописать:

```
* Поведение по брейкпоинтам:
*   >= 1260      — развёрнут (.sb)
*   760–1259     — свёрнут (.sb--collapsed) — иконки без подписей
*   < 760        — скрыт из потока, открывается оверлеем (.sb-overlay)
*
* CSS-скрытие на mobile должно жить НА УРОВНЕ ПОТРЕБИТЕЛЯ (DS не знает, куда
* Sidebar встроен). Рекомендуемое правило в потребителе:
*   @media (max-width: 759px) {
*     .app-shell > .sb:not(.sb--collapsed) { display: none; }
*   }
```

### Срочность

**Низкая** — наблюдательность.

## Что ДС делает ОТЛИЧНО (для баланса)

- **3-слойная архитектура токенов (primitive → semantic → component)** — реально удобно. Имена семантического слоя интуитивны (`--text-primary`, `--border-focus`, `--status-danger-bg`).
- **Платформенный скоупинг через media-query на `@import`** работает идеально, как только понятно про universal-компоненты. DS-файлы НЕ модифицируются — это огромный win.
- **Sidebar и Topbar CSS-классы** — чистые, без лишнего. `.sb--collapsed`, `.sb-item.is-active`, `.tb-banner` — всё на своих местах. После того, как мы написали React-компоненты по канону, всё «зажглось» сразу.
- **Brand `#0097A7`** теплый и хорошо отбивается на нейтральном фоне. Лого-mark выглядит уместно и в развёрнутом, и в свёрнутом Sidebar.
- **Component tokens с `var()` на semantic/primitive** — позволяет менять размеры через токены, не трогая компонентный CSS. На tablet/mobile это и даёт работать одним media-query.
